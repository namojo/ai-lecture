# 에이전트 팀 구성 전략

## 학습 목표

이 챕터를 완료하면 다음을 수행할 수 있습니다:

- 프로젝트 규모에 따라 에이전트 수를 결정하기
- 역할 분리 원칙에 따라 에이전트별 책임을 설계하기
- 에이전트 간 커뮤니케이션 패턴을 구성하기

## 에이전트 1개 vs 팀: 언제 무엇을 쓸까?

모든 작업에 멀티 에이전트가 필요한 것은 아닙니다. 판단 기준은 **작업의 독립성과 전문성 요구 수준**입니다.

| 상황 | 권장 구성 | 이유 |
|------|----------|------|
| 단일 파일 리팩토링 | 에이전트 1개 | 컨텍스트가 좁고 단일 관심사 |
| API 엔드포인트 추가 | 에이전트 1개 | 순차적 작업, 분리 이점 없음 |
| 전체 프로젝트 생성 | 2~3개 팀 | 프론트/백/인프라 전문성 분리 |
| 코드 리뷰 파이프라인 | 2개 팀 | 분석과 판단의 관심사 분리 |
| 대규모 마이그레이션 | 3~5개 팀 | 모듈별 병렬 처리 필요 |

> [!TIP] 판단 기준
> "이 작업을 인간 개발자에게 맡긴다면 한 명이면 충분한가, 아니면 역할을 나누겠는가?"를 기준으로 생각하세요. 에이전트 팀 구성도 동일한 논리를 따릅니다.

## 역할 분리 원칙

에이전트 팀을 구성할 때 가장 중요한 원칙은 **단일 책임(Single Responsibility)**입니다. 각 에이전트는 하나의 명확한 역할만 수행해야 합니다.

### 역할 분리 패턴

```
패턴 1: 계층 분리 (Layer Separation)
  - frontend-agent: UI 컴포넌트, 스타일링, 사용자 상호작용
  - backend-agent:  API, 비즈니스 로직, 데이터 처리
  - infra-agent:    설정 파일, 빌드 스크립트, Docker

패턴 2: 기능 분리 (Feature Separation)
  - auth-agent:     인증/인가 모듈 전체
  - dashboard-agent: 대시보드 기능 전체
  - report-agent:   리포트 생성 기능 전체

패턴 3: 파이프라인 분리 (Pipeline Separation)
  - analyzer-agent: 입력 데이터 분석 및 구조화
  - processor-agent: 핵심 로직 처리
  - reviewer-agent:  결과물 검증 및 피드백
```

> [!WARNING] 과도한 분리 주의
> 에이전트가 5개를 초과하면 오케스트레이션 복잡도가 급격히 증가합니다. 에이전트 간 데이터 전달, 순서 관리, 에러 처리가 기하급수적으로 복잡해지므로 **3개 이하를 권장**합니다.

## 네이밍 컨벤션

에이전트 이름은 역할을 즉시 파악할 수 있도록 명명합니다:

```
규칙: {역할}-agent 또는 {동사}-agent

좋은 예:
  code-analyzer     → 코드 분석 담당임이 명확
  security-reviewer  → 보안 검토 담당임이 명확
  test-generator     → 테스트 생성 담당임이 명확

나쁜 예:
  agent-1           → 역할 불명
  helper             → 무엇을 돕는지 불명
  claude-worker      → 구체적 책임 불명
```

## 에이전트 간 커뮤니케이션 패턴

멀티 에이전트 시스템에서 에이전트 간 데이터 교환은 크게 세 가지 패턴을 따릅니다:

| 패턴 | 흐름 | 적합한 작업 |
|------|------|------------|
| **순차 파이프라인** | `[analyzer] → [reviewer] → 리포트` | 코드 리뷰, 번역 검수 |
| **병렬 팬아웃** | `[orchestrator] → [front] + [back] + [test]` | 프로젝트 초기 코드 생성 |
| **피드백 루프** | `[generator] ←→ [reviewer]` 반복 | 품질이 중요한 작업 |

> [!INFO] 실무 권장사항
> 삼성 프로젝트에서는 **순차 파이프라인 + 최대 1회 피드백**을 기본 패턴으로 권장합니다. 무한 루프 방지를 위해 항상 `maxIterations` 값을 설정하세요.

## 에이전트 정의 구조

각 에이전트는 다음 요소를 명확히 정의해야 합니다:

```typescript
interface AgentDefinition {
  name: string;           // 에이전트 식별자 (예: "code-analyzer")
  role: string;           // 한 줄 역할 설명
  systemPrompt: string;   // 행동 지침과 제약사항
  inputSchema: object;    // 입력 데이터 형식
  outputSchema: object;   // 출력 데이터 형식
  constraints: string[];  // 금지 행동 목록
}
```

> [!TIP] 출력 스키마의 중요성
> `outputSchema`를 명확히 정의하면 다음 에이전트가 파싱할 때 오류가 줄어듭니다. JSON 스키마로 정의하고, 예시 출력을 시스템 프롬프트에 포함하세요.

## 요약

- **1개 에이전트**: 단일 파일, 단순 작업에 적합
- **2~3개 팀**: 역할이 명확히 구분되는 프로젝트에 권장
- 역할 분리는 계층, 기능, 파이프라인 패턴 중 선택
- 에이전트 이름은 `{역할}-agent` 형식으로 역할을 명확히 표현
- 커뮤니케이션은 순차, 병렬, 피드백 루프 세 가지 패턴 활용
- 각 에이전트의 입출력 스키마를 명확히 정의하는 것이 핵심
