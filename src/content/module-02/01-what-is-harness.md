# 하네스란 무엇인가: 에이전트, 스킬, 오케스트레이터

## 학습 목표

이 챕터를 완료하면 다음을 이해할 수 있습니다:

- 하네스 시스템의 개념과 LLM 단독 사용과의 차이
- 에이전트, 스킬, 오케스트레이터 각각의 역할
- 하네스를 도입해야 하는 시점과 판단 기준

## 하네스 시스템이란?

하네스(Harness)는 LLM의 행동을 **구조화하고 제어하는 시스템**입니다. 단순히 프롬프트를 보내고 응답을 받는 것이 아니라, LLM이 어떤 도구를 쓸 수 있는지, 어떤 순서로 작업을 수행하는지, 결과를 누가 검증하는지를 정의합니다.

> [!TIP] 실무 비유
> 하네스는 마치 **CI/CD 파이프라인**과 같습니다. Jenkins나 GitHub Actions에서 빌드, 테스트, 배포 단계를 정의하듯이, 하네스에서는 AI의 분석, 생성, 검증 단계를 정의합니다.

## LLM 단독 사용 vs 하네스 기반 사용

| 구분 | LLM 단독 (채팅) | 하네스 기반 시스템 |
|------|-----------------|-------------------|
| 작업 범위 | 한 번에 하나의 질문 | 복수의 단계를 자동 연결 |
| 도구 사용 | 사용자가 수동 복사 | 파일 읽기/쓰기, CLI 실행 자동화 |
| 품질 관리 | 사람이 직접 검증 | 리뷰 에이전트가 자동 검증 |
| 재현성 | 프롬프트에 의존 | 설정 파일로 재현 가능 |
| 협업 | 개인 도구 | 팀 전체 공유 가능 |

## 핵심 구성 요소

### 에이전트 (Agent)

에이전트는 **특정 역할과 전문성을 가진 AI 페르소나**입니다. 시스템 프롬프트, 사용 가능한 도구, 행동 규칙이 정의된 하나의 실행 단위입니다.

```
에이전트 = 역할(프롬프트) + 도구(tools) + 제약(rules)
```

예시: 코드 리뷰 에이전트, 테스트 생성 에이전트, 문서화 에이전트

### 스킬 (Skill)

스킬은 **재사용 가능한 작업 단위**입니다. 에이전트가 특정 상황에서 호출할 수 있는 미니 워크플로우로, 트리거 조건과 참조 파일을 포함합니다.

```
스킬 = 트리거 조건 + 지시사항 + 참조 자료
```

예시: `/review` 스킬, `/generate-test` 스킬, `/analyze-log` 스킬

### 오케스트레이터 (Orchestrator)

오케스트레이터는 **여러 에이전트와 스킬을 조율하는 상위 컨트롤러**입니다. 작업을 분배하고, 결과를 수집하며, 최종 판단을 내립니다.

```
오케스트레이터 = 작업 분배 + 결과 수집 + 흐름 제어
```

> [!WARNING] 과도한 구조화 주의
> 모든 작업에 하네스가 필요한 것은 아닙니다. 단순 질의응답이나 일회성 코드 생성에는 직접 프롬프트가 더 효율적입니다. 하네스는 **반복적이고, 다단계이며, 팀이 공유해야 하는 작업**에 도입하세요.

## 하네스 도입 판단 기준

다음 조건 중 **2개 이상**에 해당하면 하네스 도입을 고려하세요:

1. 동일한 프롬프트를 반복적으로 사용한다
2. 작업이 2단계 이상의 순차적 과정을 포함한다
3. 결과물에 대한 일관된 품질 기준이 있다
4. 팀원 간 AI 활용 방식을 표준화해야 한다
5. 특정 도구(파일 시스템, CLI 등)와의 연동이 필요하다

## 삼성 실무 적용 사례

```
[코드 PR 제출] → [리뷰 에이전트: 정적 분석] → [보안 에이전트: 취약점 검사]
                → [문서 에이전트: 변경 이력 생성] → [오케스트레이터: 종합 리포트]
```

## 요약

- 하네스는 LLM의 행동을 구조화하고 제어하는 시스템
- 에이전트(역할), 스킬(재사용 작업), 오케스트레이터(조율)로 구성
- 반복적/다단계/팀 공유 작업에서 하네스가 효과적
- CI/CD 파이프라인처럼 AI 작업 흐름을 코드로 관리할 수 있음
